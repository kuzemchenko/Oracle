## §R2. Граф последствий и методика отбора

Граф от одной новости — до ~100 узлов 2–4 порядка. Отбор — **воронка, не плоский
чек-лист**: порядок применения критериев критичен (дёшево-убивающее раньше дорогого).

**Центральное напряжение (иначе — ловушка Brent):** ранжирование по вероятности или силе
сцепления даёт очевидный, уже отыгранный 1-й порядок. Альфа — в конфликте критериев. Ось
ранжирования = **|неотыгранный edge| × неочевидность**, а вероятность/сила/торгуемость —
ворота-минимумы, НЕ ось.

### Ступень 0 — жёсткие ворота (бинарно, по всем узлам; провал → research/watchlist)
1. **Торгуемость** — есть ли §9-разрешимый инструмент (динамич. реестр / добор EODHD). `is_sealable`.
2. **Ликвидность/выразимость** — мин. объём + выразимо ли направление (шорт/опционы). `costs.yaml`.
3. **Инвестируемое окно** — лаг в досягаемом окне (дни–месяцы, не 3 года). Дальше → watchlist.
4. **Разрешимость §9** — формулируется как фальсифицируемый прогноз (порог+дата). `validate_resolvable`.
**Ярус сцепления (A/B/C) — НЕ ворота (директива 20.06).** Нехватка истории/эмпирики НЕ отбрасывает
событие — иначе вылетают именно неочевидные дальние узлы, суть продукта (откат к Brent-ловушке).
Ярус снижает РАНГ через надёжность в пред-ранге, но узел ОСТАЁТСЯ в потоке с research-меткой.
Что делать с ярус-B/C на этапе SEAL — РЕШЕНО 20.06 (Вариант 2, см. §R3 «Два трека seal»):
ярус A (established) → денежный трек §11; ярус B/C с инструментом → ПРОВИЗОРНЫЙ форвард-трек.
НЕ критерий отбрасывания.

### Ступень 1 — дешёвый пред-ранг (мультипликативно; сортировка → топ-K)
Мультипликативно: фактор около нуля убивает узел.
- **Сила сцепления** = |Π gain| по пути (беты/экспозиции). `compose_chain`.
- **Надёжность пути** = Π reliability (наказывает длинные спекулятивные цепи).
- **Сигнал над шумом** = amplitude / σ_горизонта (должен пробить шумовой пол).
- **Изоляция сигнала** = R² узла на источник + число коллизий (сколько каскадов бьют в инструмент).

Реализация B1 (`mathlib/graph_select.py`, ✅ 20.06): discovery-ранг = **snr × изоляция** = ВОЗМОЖНОСТЬ.
**Надёжность УБРАНА из ранга** (директива 20.06): её компаундинг по ярусу-C (0.2^хопы ≈ 0.008)
хоронил дальние неочевидные узлы — суть продукта (NUE 4-го порядка: дно → 2-е место). Надёжность →
МЕТКА уверенности + ворота ДЕНЕЖНОГО трека (route_tracks), НЕ буритель ранга. «Измеренность» связи
входит через изоляцию (R² терминала на корень), магнитуду неотыгранного — через snr (|edge|/σ_h).
Изоляция без истории → структурный дефолт «неизмеримо» (§R2.1 — не выдумка, П8); коллизия → счётчик
+ штраф 1/(1+λk) (§R2.4 — неоднозначность подтверждение-vs-перегрев во флаге).

### Ступень 2 — дорогой анализ (топ-K 5–10; математика + LLM + слепой суд)
- **Неотыгранный edge** = amplitude − реализованное. `cascade_edge`. (Заголовок.)
- **Вероятность реализации** = надёжность пути × направленная P, калибр. по §R3.
- **Тайминг-распределение** — окно реализации, не точка-лаг (§R3 калибрует).
- **Неочевидность** — насыщенность новостей по цели + позиционирование. `c_non_obviousness`.
- **Кто продаёт и почему неправ + анти-манип** — §8, поведенческий ответ (§R4).
- **Издержки выражения** — шорт/borrow-прокси/skew/спред. `costs.yaml` + §R4.

**Отбор инструмента** выпадает сам: ворота торгуемости + изоляция/сигнал-над-шумом выбирают
лучший инструмент на событие; события без приличного инструмента → research-дайджест.

**Сборка:** ворота → P1×P2×P3×P4 → топ-K → взвешенный §7-скоринг → слепой суд.

### Открытые исследовательские задачи (копать при сборке)
1. Изоляция сигнала без истории — структурный фолбэк + «неизмеримо → штраф/research», не выдумка (П8).
2. Компаундинг вероятности по хопам — определяется слабейшим звеном. **Решено 20.06: жёсткого
   лимита глубины НЕТ; гейт seal — порог надёжности пути.** Это совпадает с существующим
   `compose_chain.sealable_path` (запечатываемо только если ВСЕ звенья ярус A + перенос установлен) —
   глубина отсекается сама, т.к. длинные цепи редко полностью ярус-A. Значение порога надёжности
   тюнит петля (предложением, §R6), не фиксированное правило.
3. Квантификация «рационального» сцепления (ярус B) без цены — экспозиция %/опер.рычаг/доля; нет источника → П8.
4. Коллизия каскадов — подтверждение vs перегретый консенсус (независимость драйверов).
5. Калибровка трёх оценщиков (edge/вероятность/тайминг) на разнородных событиях (§R3).

## §R3. Модель оценки для уникально-событийного режима

Проблема: повторная калибровка «по инструменту» (walk-forward бета пары, N≥30 повторов
одного прогноза) НЕ переносится на уникальные события графа. Единица оценки сдвигается на
три повторяемые сущности:

1. **Рёбра графа** — перенос инстр→инстр повторяется across событий → калибруемо (ярусы
   A/B/C уже это кодируют: A=история, B/C=форвард/research). Это повторяемая подложка.
2. **Калибровка вероятностей системы** — Brier / диаграмма надёжности работают на
   разнородных уникальных событиях (метеоролог калибруется на неповторяющихся днях). Статистика
   группируется **по корзинам уверенности**, не по инструменту. §10 N≥30 переосмысляется: 30
   прогнозов в корзине «65–75%», а не 30 на одном тикере.
3. **Мета-калибровка edge-оценщика** — из движений, названных «не отыграно», доля реально
   сдвинувшихся. Аналогично — **калибровка тайминг-оценщика**: попало ли реализованное в окно.

**Брандмауэр П16:** бэктест на истории — ТОЛЬКО детерминированной машинерии (граф/рёбра/
edge/тайминг-оценщик — это §23.1-легально). Качество выбора идей LLM на истории до cutoff —
НИКОГДА. Это разделяющая линия для петли §R6.

### Два трека seal (РЕШЕНО 20.06 — Вариант 2)

Каскад с ярус-B/C основой НЕ выбрасывается и НЕ остаётся «мёртвым research» (это был бы круг:
не запечатали → нет исхода → гипотеза навечно гипотеза). Вместо этого — ДВА запечатываемых трека:

- **Денежный трек** — только ярус A (established, перенос установлен). Считается к воротам §11,
  входит в денежный Brier. Строгий.
- **Провизорный трек** — ярус B/C с §9-инструментом. Запечатан форвардно (timestamp+hash, П16 —
  реальный фальсифицируемый прогноз с меткой «основа=гипотеза»), разрешается, копит СВОЙ Brier.
  **К §11 не приближается, в денежный Brier не подмешивается** (герметичная метка/раздельный расчёт).
  ВЫПУСК в денежный трек — когда класс гипотез накопил N≥30 разрешённых и калиброван по корзинам
  (§R3 выше). Так гипотеза ЗАРАБАТЫВАЕТ основу форвардом (цены её не дадут — §R2-находка).

«Запечатать» здесь = записать проверяемый прогноз и обязаться сверить, НЕ «поставить деньги»
(деньги — отдельные ворота §11). Узлы без §9-инструмента/неразрешимые → research-дайджест (не seal).
Дисциплина: провизорный трек держать ОТДЕЛЬНО от денежного (метка в журнале + раздельный Brier),
иначе гипотезы протекут в §11. Реализация — Этап B3.

## §R6. devops-петля-предложитель (Карпатый, автономия 0)

**Петля ничего не применяет сама.** Каждый цикл выдаёт: diff + отчёт надёжности (диаграмма §R3) +
контрфактический протокол (§11.1). Человек применяет руками — это «сохранить понимание».

- **Тюнит (бэктестом на детерминированной машинерии, P16-чисто):** широту скана, состав рёбер
  графа, построение графа/волн, здоровье кода, приоритизацию ветвей.
- **Только предлагает (на подпись):** веса §7, критерии, пороги, денежные ворота, kill-критерий.
- **Цель/стоп:** качество террейна (поток высоко-edge неочевидных узлов + покрытие графа) как
  быстрый сигнал; пулированный Brier по корзинам уверенности (§R3) — медленный честный гейт.
  Не «крутись пока не достигнешь Brier» (это переобучение под форвард-трек, нарушение П16).

  *Гейт:* промпты синхронны, П8 в каждом.
- **Этап E — модель оценки (§R3).** Диаграмма надёжности по корзинам + калибровка edge/тайминг.
  *Гейт:* harness считает калибровку на разнородных запечатанных, P16-чисто.
- **Этап F — петля-предложитель (§R6).** Бэктест детерм. машинерии → diff + отчёт + контрфактика.
  *Гейт:* цикл выдаёт предложение, НИЧЕГО не применяет сам.
- **Этап G — интерактивный пульт-рассуждение (§R9).** Telegram-Дирижёр как инструментальный агент:
  интроспекция (read-tools) + ad-hoc проверка идеи (полный контур) + propose-only изменения.
  Зависит от B/C (каскадная машинерия для проверки идей). *Гейт:* в чате можно получить
  пояснение логики по факту (с цитатой расчёта), проверить идею с новости полным разбором, и
  получить diff-предложение — при НУЛЕВОЙ мутации весов/лимитов/журналов.

Где есть LLM-стоимость — живая сверка только по go пользователя (ворота §11/П16).

## §R9. Интерактивный пульт-рассуждение (Telegram)

Цель: дать в Telegram то же, что есть в интерактиве Claude Code — думающего собеседника,
заземлённого на ЖИВУЮ систему, с которым можно рассуждать про события/цифры/аргументы,
требовать пояснения логики и проверять свежие идеи с новостей. Реализация: `bot_chat.py`
дорастает из «ответчика по брифингу» в **инструментального агента** (LLM в петле с
инструментами), роль `conductor` (Opus).

### Три способности

**1. Интроспекция-заземление.** Набор READ-инструментов: расчёт амплитуды/edge по узлу,
причина отсева идеи на этапе воронки, Brier по корзинам, срез конкретного агента, протокол
прогона, число из журнала. Дирижёр объясняет **по факту с цитатой расчёта** (П8), а не по памяти.

**2. Ad-hoc «проверь идею» (свежая, с новости).** Вход: новость / «тикер + тезис». **Всегда
полный состязательный контур** (решение 20.06): картограф → `mathlib/cascade` (амплитуда/edge) →
поведенческий слой (§R4) → тайминг + неочевидность → генератор/критик/слепой судья (П10) →
§8-вердикт с research-меткой (seal только если проходит гейт §9/надёжности). Стоит ≈ event-first
(~$5), под воротами §11 (exit_code 3 блокирует). Отличие от `/debate`: тот по УЖЕ выданной идее,
этот — с нуля.

**3. «Поменять» = propose-only + read (решение 20.06).** Дирижёр из чата читает что угодно,
запускает проверки/прогоны, готовит **diff конфига на твою подпись** — но НЕ мутирует сам:
веса §7, `limits.yaml`, рубрику, запечатанные журналы неприкосновенны. Тот же брандмауэр, что у
петли §R6. Журналы дополнительно прикрыты хуком `guard_journal` (блокирует запись в
`predictions.jsonl`, П16) — забор автоматический.

### Анти-сикофантность (обязательно)
«Рассуждать об меня» работает, только если Дирижёр может **возразить по существу**: «уже в
цене», «нет данных, чтобы это утверждать», «ты в ПОЗДНО». Тот же хребет П8 — не эхо твоих мыслей,
а проверка. Промпт `conductor` несёт это явно.

### Стоимость
Каждая реплика и каждая проверка идеи пишется в `costs.jsonl` (ворота §11). Полный разбор по
умолчанию (решение 20.06) → бюджет-осознанность: при подходе к потолку Дирижёр предупреждает,
exit_code 3 блокирует. Дешёвой деградации нет — это твой явный выбор качества над ценой.
