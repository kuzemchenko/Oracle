## НЕИЗМЕНЯЕМЫЕ РАМКИ ПРОГРАММЫ (граница, нарушение = стоп этапа)

1. **Планка суда 3.0, рубрика (v1.4), ворота §11/270/KILL, запечатанные журналы

2. **Все новые числа — детерминированный код с walk-forward-валидацией (§23.1, Инв#6).**
   LLM не оценивает величины: LLM предлагает карту/механизм, код меряет.
3. **«Искать дальше» разрешено ТОЛЬКО как перебор по инструментам при зафиксированном
   событии и зафиксированных порогах.** Перебор по порогам до пролезания (df, q, планка,

   фиксируются ДО прогона по walk-forward-отчёту с провенансом; смена порога = новый
   walk-forward-отчёт + запись причины, не реакция на «не пролезло».
4. **Одна сессия = один этап = один /goal.** Этапы без зависимости можно вести
   параллельными сессиями (Д1 ∥ Д2 ∥ Д3 ∥ Э4-каркас). Слияние — только через master
   после гейта этапа.

# ЭТАП Д2 — аудит калибровочного трека (независим, срочный)

**Тег:** `se-d2` (база `se-d2-base`). **Зависимости:** нет. **Бюджет:** ≤ $5.

## Факт (из SYNC 13.07, §4.6)
252 разрешённых калибровочных исхода, hit-rate **36.1%** при заявленной вероятности
каждой печати **0.500**, calibration band 46.21 п.п. Контур полностью детерминирован:
`orchestrator/calibrate.py` (3 прогноза на актив: пороги {close, ±0.5σ_H}, H=5),
гауссова база без дрейфа, сверка `orchestrator/resolve.py` + `mathlib/outcomes.py`.
При честной монетке ожидание hit ≈ 50%; 36.1% на N=252 — систематика, а не шум
(биномиальный p << 0.01). Направление подозрения — конвенции, не рынок.

## Работа
1. **Построчный аудит конвенций** (только чтение кода + пересчёт): направление
   (above/below против знака смещения порога), размещение порогов (±0.5σ_H от какого
   close; adjusted vs raw — прецедент F0#8), логика сверки (строгие неравенства F-фикса
   «равенство ≠ успех»; какой бар берётся как исход при выходных/пропусках; resolve_by
   и часовой пояс), согласованность σ_H (какая σ и на каком окне — прецедент долга B4(в)
   «две σ в разных местах»).
2. **Диагностическая переоценка исходов — В ОТЧЁТ:** независимый пересчёт всех 252
   исходов из сырых котировок `storage/oracle.db` отдельным скриптом; разложение
   расхождений по причинам. **Журналы неприкосновенны** — `outcomes.jsonl` не
   редактируется и не дополняется задним числом. **Табло §15 показывает ДВА ряда
   (решение владельца 13.07, Вопрос 2):** официальный (как записано в журнале) +
   диагностический с явной пометкой «после найденной ошибки»; официальный ряд
   первичен, диагностический — только чтение из отчёта.
3. **Классификация первопричины:** (а) **баг** (конвенция/сверка врёт) → фикс ВПЕРЁД
   (новые печати/сверки правильные, старые не переписываются) + regression-тест,
   воспроизводящий ошибку на зафиксированном примере; (б) **неадекватность базы**
   (бездрейфовый гаусс систематически смещён на этих порогах/горизонте) → замена базы
   с walk-forward-валидацией (§23.1: OOS-частоты против заявленных вероятностей),
   провенанс в коде/конфиге.
4. **Следствие для П-1 зафиксировать:** `knowledge/prob_shrink.yaml` (λ=0.0, p0=0.431)
   подогнан по исходам, среди которых калибровочные конвенции могли быть грязными.
   В `ops/refit_prob_shrink.py` — комментарий-зависимость: «p0/λ подлежат
   пере-подгонке после накопления чистых пост-Д2 исходов». **Решение владельца
   13.07 (Вопрос 3): ЕСЛИ Д2 подтверждает баг сверки — авто-refit цикла 01.08
   ставится на паузу (один цикл, правка crontab при закрытии Д2 с записью причины);
   пере-подгонка возобновляется штатно по чистым исходам. Если бага нет — цикл
   01.08 идёт как подписано 09.07.**

# ЭТАП Д3 — условная амплитуда: event-study вместо безусловной беты (тяжёлый)

**Тег:** `se-d3` (база `se-d3-base`). **Зависимости:** нет (параллелен Д1/Д2).
**Бюджет:** ≤ $10.

## Факт (из SYNC 13.07, §3.3/§4.3/§4.5)
`mathlib/cascade.py:node_sensitivity` — безусловная OLS-бета на дневных закрытиях за
годы истории. На боевых кандидатах r² = 0.001–0.23; эмпирические лаги все = 0
(`knowledge/causal_links.yaml`: дневные ETF синхронны, библиотека фактически
ненаправленная); ярус A структурно недостижим для неочевидных пар; вердикты суда —
«статистический перенос отсутствует (r²=0.0)», «ход меньше шума». Безусловная бета
за годы усредняет перенос по спокойным дням, в которых связи нет, — событийный
перенос она не видит по построению.

## Работа
1. **Условный оцениватель в mathlib (честная зона §23.1):**
   - выделение **эпизодов шока источника** — **ЕДИНЫЙ порог для всех пар (решение
     владельца 13.07, Вопрос 4):** |shock| ≥ 0.5·σ_ист·√окна (консистентно с
     активацией B4), зафиксирован ДО оценки целей; в walk-forward-отчёте —
     ОБЯЗАТЕЛЬНАЯ проверка устойчивости выводов к сдвигу порога (±диапазон),
     не подбор порога (рамка 3); индивидуальные пороги — только если единый
     провалит проверку устойчивости, отдельным решением;
   - для каждой пары (источник → цель): **распределение отклика цели** по окнам лагов
     (0…L дней) в эпизодах против базового распределения цели вне эпизодов;
     условный gain, доверительный интервал, эффект-статистика — детерминированно;
   - **walk-forward:** эпизоды train-окна → проверка на OOS-эпизодах; пары без
     устойчивости — «не установлено» (П8), не пин;
   - **N_эпизодов = sample size ярусов честности:** мало эпизодов → ярус C
     («механизм, не подтверждён»), достаточно + OOS-устойчивость → кандидат в ярус A
     по условному переносу; маппинг N→ярус фиксируется в отчёте калибровки, не в голове.
2. **Интеграция в `mathlib/cascade.py` ОТДЕЛЬНЫМ путём с провенансом:** узел несёт
   оба измерения — `sensitivity_unconditional` (существующая бета, остаётся для
   сравнения и совместимости) и `sensitivity_conditional` (новая, с N_эпизодов,
   CI, лагом-окном, walk-forward-меткой). Какое из них кормит ворота/ранг —
   решение этапа Э4(в)/(г), НЕ Д3: Д3 только добавляет честное измерение, ничего
   в боевом пути не переключая.
3. **Калибровочный драйвер** `ops/calibrate_conditional.py` (или расширение
   `calibrate_sensitivities.py`): отчёт в `ops/reports/`, YAML-артефакт в `knowledge/`
   со статусом «справочный» (движок считает живьём, как сейчас — прецедент FГ2 §9.1).

## Гейт Д3
- Walk-forward-отчёт опубликован. Контроли: **USO→BNO остаётся сильной** (условный
  оцениватель не ломает известную связь); на **событийных парах** из боевых журналов
  (примеры из вердиктов суда: XOM→EEM r²=0.0, новость→FRO/STNG/DHT) показан условный
  перенос там, где безусловный r²≈0, ЛИБО честно показано его отсутствие — оба исхода
  легитимны, гейт требует, чтобы измерение состоялось и было walk-forward-чистым,
  а не чтобы перенос «нашёлся» (рамка 3).
- **Characterization-тест: calibrate-режим байт-в-байт не изменился** (печати
  calibrate не зависят от нового оценивателя); event_first/replay протоколы без
  переключения тоже не меняются (новые поля — только добавлены).
- stage-review + Codex ПРОЙДЕНЫ; тесты зелёные (юнит-тесты оценивателя на синтетике:
  вшитый эпизодный перенос находится, шум — не находится).

# ЭТАП Э4 — «Перебор мира»: инфраструктура перечисления (параллелен Д1–Д3)

**Тег:** `se-e4` (база `se-e4-base`). **Зависимости:** каркас (а)(б)(д)(е)(ж) — нет;
пункты **(в)(г) гейтуются только после Д3**. **Бюджет:** ≤ $15 (включая mock-гейт).

## Факт (из SYNC 13.07, §3.1/§4.2/§4.5)
Генерация схлопнута до одного вызова картографа: `max_map=10`, фактически 3–7 пар
«новость→компания» в день; отказ инструмента (нет данных/не sealable) убивает событие
целиком — второй кандидат не пробуется; §6 эт.2 канона (30–50 кандидатов) потерян при
переходе на event-first (SYNC §7.1). Библиотека B4 — 22 ребра, у 21 из них N=1.

## Работа
- **(а) LLM-карта мира события:** вход — событие (кластер/шок); выход — карта
  СЕГМЕНТОВ и МЕХАНИЗМОВ влияния (порядки 1–4, направление давления, канал переноса),
  **НЕ тикеры**. LLM предлагает структуру, числа не оценивает (рамка 2). Роль
  generator (opus), отдельный суб-потолок `world_map` (см. бюджет). Карта
  журналируется в протокол прогона с провенансом.
- **(б) Детерминированный скрин сегмент→инструменты:** по каждому сегменту карты —
  полный список ликвидных торгуемых инструментов: EODHD screener/Tier0-фундаментал
  (сектор/индустрия/выручка), фильтр ликвидности из `universe.yaml`
  (`min_avg_daily_volume`), sealable-гейт через `universe_resolver`. Цель —
  **100–300 инструментов на тектоническое событие**. Наличие screener-эндпойнта
  проверяется живьём в начале этапа; фолбэк — Tier0-фундаментал по спискам сектора.
  **Добор истории — БЕЗ дневного потолка (решение владельца 13.07, Вопрос 5):**
  качаем всё сразу; при исчерпании дневной квоты EODHD — алерт в бота через
  существующий канал (`cron_alert`/notices), чтобы исчерпание не было тихим.
  Код, не LLM.
- **(в) Скоринг всех пар условным движком** (ПОСЛЕ Д3): для каждой пары
  событие-источник → инструмент — условная чувствительность/амплитуда/edge
  (`sensitivity_conditional`), с **BH-поправкой ВНУТРИ события** (m = число
  инструментов перебора этого события; те же mathlib/fdr-процедуры) — перебор сотен
  пар без множественной поправки запрещён (рамка 3: это и есть манипуляция гейтом).
- **(г) Ранг пары** (ПОСЛЕ Д3) = условная чувствительность × неотыгранность
  (attention-датчик П1 + пройденный ход к расчётной амплитуде) × бонус порядка 2–4
  (глубина узла карты). Детерминированная формула с зафиксированными коэффициентами
  (провенанс: walk-forward где измеримо, иначе объявленный прайор с пометкой —
  прецедент П2б). Ранг НЕ трогает скоринг §7/веса §30.6 — он выбирает, ЧТО подать
  в существующие ворота/суд.
- **(д) Возврат по причине:** отказ кандидата классифицируется кодом:
  **инструмент-специфичный** (нет истории/нет ликвидности/не sealable/нет данных
  скрина) → берётся следующий по рангу, кэп попыток на событие + бюджет события
  (константы в конфиге, зафиксированы до прогона); **событие-специфичный**
  (карта пуста/шок не подтверждён/BH внутри события всё срезал) → стоп по событию
  с журналом причины. Каждый возврат — строка в протоколе (категория, кандидат,
  причина дословно) → attrition-таблица становится восстановимой (закрывает дыру
  логирования из SYNC §4.2: `отсев_по_критериям` сейчас пуст).
- **(е) Персистентность тектонических тем:** карта мира события живёт дольше дня —
  еженедельный пере-скрин сегментов активных карт (новые инструменты, изменившаяся
  ликвидность), с провенансом и сроком жизни карты. **Расписание: воскресенье 09:00
  (решение владельца 13.07, Вопрос 6)** — после воскресного блока 08:00/08:30,
  без сообщений владельцу.
- **(ж) Кормление библиотеки B4 — АВТОМАТИЧЕСКИЙ append (решение владельца 13.07,
  Вопрос 7):** каждая пара с механизмом (независимо от исхода ранга) добавляется
  в библиотеку форвард-теста append'ом с провенансом (событие, карта, дата,
  механизм); список добавленного за неделю — в воскресном отчёте §25. Плохие рёбра
  выбраковывает форвард-статистика; **промоушен ребра в «доверенные» — по-прежнему
  ТОЛЬКО рукой владельца** (`promote_edges.py --apply`). Сейчас 22 ребра, у 21 N=1;
  цель программы ≥100. Пересечение с долгом B4(а) (псевдорепликация корма
  промоушена) — дедуп эпизода в корме входит в объём этого пункта: см. секцию
  «Не в этой программе».

## Гейт Э4
- **Mock end-to-end на историческом событии `ai_power`** (данные в БД, карта известна
  независимо): перечислено 100+ инструментов после скрина; скоринг+BH внутри события
  посчитаны (после Д3); ранг построен; retry-логика (д) покрыта тестами (оба класса
  отказов + кэп попыток + бюджет события); бюджет события закэпован и виден в протоколе.
- Пункты (в)(г) гейтуются только после Д3 (walk-forward условного движка); каркас
  (а)(б)(д)(е)(ж) можно закрыть промежуточным stage-review раньше, финальный гейт
  этапа — по полному mock.
- Ни одна печать/боевой путь не изменены (Э4 строится рядом; переключение — Э5).
- stage-review + Codex ПРОЙДЕНЫ; тесты зелёные.

# ЭТАП Э5 — интеграция в боевой контур (после Д1+Д2+Д3+Э4)

**Тег:** `se-e5` (база `se-e5-base`). **Зависимости:** Д1, Д2, Д3, Э4 закрыты (теги).
**Бюджет:** ≤ $25 (≥3 live-дня в существующем капе $5.00 + разбор).

## Работа
Боевой крон 09:00 переключается на путь перебора (карта мира → скрин → условный
скоринг+BH → ранг → существующие ворота графа): `--vet`/`--seal`/`--deep`, треки
money/провизорный/дайджест, fail-closed маршрутизация money (`_money_kind`),
планка 3.0, П-1-сжатие — **без изменений**. Полный live-прогон + дифф протоколов
до/после (тот же день, старый и новый путь) в отчёт этапа. Переключение — одним
явным коммитом с возможностью отката (старый путь остаётся вызываемым).

**Правило подачи (решение владельца 13.07 о шуме, обязательное условие этапа):**
в Telegram — максимум ОДНО сообщение-дайджест в день с топ-3–5 идей + редкие
money-карточки после суда + алерты поломок; всё остальное — только по явному
запросу («дай идеи», /session). Потолок закрепляется кодом слоя показа
(`bot_reports`/`presentation`) и автотестом; расширение внутреннего перебора
(сотни пар) НЕ увеличивает исходящий поток.

## Гейт Э5
- **Окно 10 торговых дней: не меньше 3 дней, когда money-кандидаты дошли до суда
  (решение владельца 13.07, Вопрос 9).** Суд собирается; вердикты — какие
  заслужены: РАЗБИТА — тоже работа суда, квоты идей нет. Этап не «висит» из-за
  тихого рынка, и соблазна дотягивать кандидатов не возникает.
- Правило подачи работает: за окно гейта ни одного дня с более чем одним
  дайджест-сообщением (проверка по журналу бота), потолок покрыт автотестом.
- Attrition-таблица прогона показывает перебор: сотни пар на входе, отсев по
  именованным причинам на каждой ступени (скрин/BH/ранг/ворота/суд) — категории
  из Э4(д) в протоколе.
- **Ноль нарушений П8** за три дня (проверка протоколов: каждый факт со ссылкой,
  «нет данных» — честные категории, никакой выдуманной величины).
- Бюджет прогона в капе $5.00; дифф протоколов до/после в отчёте.
- stage-review + Codex ПРОЙДЕНЫ; тесты зелёные.

# МЕТРИКИ УСПЕХА ПРОГРАММЫ (наблюдает петля §25, пороги — не квоты)

1. **FDR жив:** событийные дни дают статистические сигналы после FDR; тихие — нет
   (по списку дней, зафиксированному до сравнения).
2. **Калибровочный трек в band ±10 п.п.** на новых (пост-Д2) исходах по мере накопления.
3. **≥1 пара яруса A по условному переносу** (walk-forward-установленная, не назначенная).
4. **Money-кандидаты в суде ≥3/нед** (кандидаты ДОХОДЯТ до суда; вердикт — дело суда).
5. **Библиотека B4 ≥100 рёбер** (с провенансом пополнения из Э4(ж)).

Метрики — диагностика программы, НЕ гейты выдачи и НЕ основания трогать пороги
(рамки 1 и 3). Недостижение метрики → разбор §25 → предложения на подпись.

# ВНЕ РАМОК ПРОГРАММЫ

- Смягчение любых порогов (планка 3.0, q=0.1, пороги активации B4, ворота графа).
- Ручная правка λ/p0 (`knowledge/prob_shrink.yaml` пишет только штатный refit-цикл).
- Квоты идей, «гарантированная выдача», любое давление на суд.
