# МАСТЕР-СПЕЦИФИКАЦИЯ «ОРАКУЛ» v10 FINAL (полный контур, утверждена)
## Мультиагентная система анализа новостного фона и инвестиционных решений

Документ самодостаточен: консолидирует v1–v6 без потери деталей (сверено аудит-чеклистом). Решение пользователя: строится сразу ПОЛНЫЙ контур — все агенты с первого дня, без поэтапного ввода. При этом контуры честности сохранены полностью (форвард-тест, бумага до денег, kill-критерий): они защищают не от сложности, а от самообмана. Вместо поэтапного A/B-ввода агентов вклад каждого измеряется абляцией (§11.1).

# ЧАСТЬ I. МИССИЯ И ПРИНЦИПЫ

## 1. Миссия
Измеримо повышать качество инвестиционных решений пользователя в его круге компетенции: генерировать **неочевидные** идеи (long/short) на основе каскадных последствий 2–4 порядка, ещё не отыгранных рынком, при строгом контроле риска и честной статистической самопроверке. Система рекомендательная: сделок не совершает, последнее слово — за человеком. Рамка каждого отчёта: «персональный исследовательский инструмент, не инвестиционная рекомендация, не для передачи третьим лицам».

## 2. Целевая функция
Максимизировать риск-взвешенную доходность портфеля рекомендаций **сверх бенчмарка** (индекс + безрисковая ставка) = f(калиброванная вероятность успеха; асимметрия выигрыш/проигрыш net издержек; неочевидность; надёжность данных; контролируемость риска) при ограничении максимальной просадки. Дополнительная метрика: **предотвращённые ошибки** — плохие ставки, от которых система удержала; самое прибыльное решение большинства дней — не сделать плохую ставку. Обучение ведётся до достижимого **плато** целевой функции на out-of-sample — «идеальность» определена как плато, а не бесконечная подгонка.

## 3. Принципы (П1–П16)

**П1. Мультиязычность.** Новости на разных языках и в разных странах; локальные источники опережают мейнстрим на часы и дни. Теги: язык, страна, тип источника (СМИ/соцсеть/форум/официоз), время.

**П2. Сантимент через частоту слов.** Количественно: частоты слов, плотность ключевых слов в поиске, индексы слов (Google Trends и аналоги), n-gram-динамика, тональность контекстов, всплески упоминаний.

**П3. Жизненный цикл феномена.** Стадии: зарождение (нишевые источники) → разгон (выход в мейнстрим) → пик → сдувание. Аппроксимация кривой популярности и её производной; определение закона развития (S-кривая, экспонента, пузырь).

**П4. Эффект толпы / «индикатор домохозяек».** Вход последних розничных участников (массовые упоминания в неспециализированных медиа, ролики «как купить X для начинающих», взрывной рост розничных счетов) = маркер вершины, «куплено на хаях». Сигнал строго контрциклический: повод для short/выхода, не входа. Научное соответствие — стадия эйфории по Киндлбергеру/Мински.

**П5. Неочевидность.** Следствия 1-го порядка отыгрываются HFT и мейнфреймами инвестбанков мгновенно — обыграть их невозможно, на этом поле не играем. Специализация — каскады 2–4 порядка: «событие → удорожание X → давление на отрасль Y → выгода для Z». Каждая идея проходит отдельную оценку на неочевидность; очевидное отбраковывается.

**П6. Credibility источников и прогнозистов.** Для каждого внешнего прогноза: track record автора (что сбылось/нет), репутация и экспертиза в данной области, спектр существующих мнений → консенсус → калиброванная вероятность.

**П7. Калиброванные вероятности.** Журнал всех вероятностных оценок; регулярная сверка: из событий с оценкой 70% должно сбываться ~70%. Метрика — Brier score по корзинам вероятностей. Раскалиброванные агенты перенастраиваются.

**П8. Нулевые выдумки (критический).** Ставки высоки — система никогда ничего не выдумывает. Нет данных → обязательный ответ «не знаю / нет данных» (поощряемый исход любого агента). Каждый факт — со ссылкой на источник, каждое число — с расчётом или первоисточником. Валидатор блокирует неподтверждённое на уровне архитектуры.

**П9. Асимметрия риска.** Только асимметричные ставки net издержек (пример: +40%/−20% с учётом вероятностей). Симметричные «выиграть 100% / потерять 100%» отбраковываются.

**П10. Состязательность разных моделей.** Модель, которая сама генерирует и сама проверяет, будет «химичить» — подспудно подтверждать себя. Генератор, критик и судья — модели РАЗНЫХ семейств (через OpenRouter); суд слепой, порядок аргументов рандомизирован, оценка по рубрике.

**П11. Стабильная калибровка.** Изменения весов — только статистически обоснованные, с регуляризацией шага, на достаточной выборке. Запрещены «дурные» скачки весов после единичных промахов и блуждание настроек.

**П12. Решение о риске — за человеком.** Система показывает: что можно заработать, что потерять, вероятности, способы балансировки (опционы, хедж, размер позиции) — и активно **рекомендует уменьшение риска**, когда он избыточен. Решение принимает пользователь в рамках пре-коммитмента.

**П13. Своевременность.** Видимая всем новость почти всегда уже в цене: к моменту публичных лент ставку сделали узнавшие раньше, дальше потенциал — вниз. Каждая идея получает вердикт стадии входа: РАНО / ВОВРЕМЯ / ПОЗДНО / ЛОВУШКА.

**П14. Защита от манипуляций.** Инсайдеры намеренно возят рынок туда-сюда, выбивая стопы и ловя «глупцов». Каждая идея обязана ответить судье: «Почему эта возможность ещё существует? Кто продаёт нам и почему он неправ?» Нет внятного ответа, чьей ошибкой оплачен наш выигрыш, — нет идеи.

**П15. Научное спокойствие.** Поведенческие выводы — только через именованные опубликованные эффекты с измеримыми в данных прокси. Нельзя измерить — честное «нет данных». Никаких «кажется, толпа нервничает».

**П16. Честная проверка edge.** Оценка качества LLM-прогнозов на истории до cutoff моделей запрещена (модель помнит будущее). Качество доказывается форвард-тестом с запечатанными прогнозами. Kill-критерий записан заранее. (Что МОЖНО честно калибровать на истории — см. §23.)

# ЧАСТЬ II. АРХИТЕКТУРА

## 4. Реестр агентов
ВСЕ агенты строятся сразу, в полном составе. Порядок сборки определяется только инженерной зависимостью (§24): данные → математика → агенты → дебаты. Вклад каждого агента измеряется абляцией (§11.1), его вес в консенсусе — меритократический (§5.3). Правило для всего блока B: суждения «историков», «циклиста» и «агента примет» НЕ имеют прямого пути в консенсус — только через критику в состязательном контуре.

| Котировки | Исторические и текущие цены: обязательны для тайминга, сверки прогнозов, бенчмарка и калибровки §23 |

## 5. Дирижёр и иерархия решения
Дирижёр НЕ имеет собственного мнения о рынке (иначе — ещё один предвзятый голос с административной властью). Функции:
1. **Маршрутизация** по воронке; созывает не всех агентов всегда — дешёвые фильтры раньше дорогих дебатов; ведёт бюджет токенов прогона (limits.yaml).
2. **Поле суждений** в стандартном формате: вывод + вероятность + уверенность + данные-основания + что неизвестно. Без оснований — возврат (П8).
3. **Меритократические веса:** вес голоса = доказанная форвард-точность агента по данному ТИПУ вопросов (из дашборда). Новичок — с минимальным весом, зарабатывает статистикой. Веса видимы, версионируются. На старте — равные веса (форвард-статистики ещё нет, равенство честнее выдуманных весов); меритократия включается по мере накопления статистики.
4. **Карта противоречий:** расхождения НЕ усредняются молча — эскалация в дебаты именно по точке расхождения; неразрешённое противоречие открыто понижает уверенность итога.
5. **Синтез:** экстремизация агрегированных независимых оценок с поправкой на их корреляцию; школам подаются частично непересекающиеся срезы данных; высокая корреляция оценок → почти без экстремизации (иначе ложная уверенность).
6. **Вето только процедурное:** остановить идею за нарушение процедуры (нет источников; не пройдена антиманипуляция; нет ответа «кто продаёт нам») может; менять содержание суждений — нет.
7. **Протокол прогона:** кто что сказал, с какими весами вошло, где противоречия, почему итог таков — сырьё для разбора промахов.

**Иерархия (зашита в коде):** агенты → Дирижёр (поле + противоречия) → состязательный контур → Дирижёр (синтез вероятности) → Риск + Портфель (их «нет» перебивает любой энтузиазм поля) → пользователь (П12).

## 6. Воронка отбора
Никогда не берётся первая попавшаяся идея.
- **Этап 1. Широкий скан:** 200–500 сырых сигналов (события, тренды, аномалии частот). **FDR-контроль:** аномалия считается сигналом только при q-value < 0.1 по процедуре Бенджамини–Хохберга относительно фоновой дисперсии метрики; порог хранится в config и калибруется на истории (§23). Без этого при 500 проверках в день система «найдёт» закономерности в шуме гарантированно.
- **Этап 2. Кандидаты:** 30–50 идей; каждая школа обязана выдвинуть свои (разнообразие точек зрения).
- **Этап 3. Грубая фильтрация → ~15.** Стоп-фильтры: очевидность; нет данных/непроверяемые источники; вне круга компетенции при среднем потенциале; манипуляционный балл выше порога; тайминг ПОЗДНО/ЛОВУШКА без контр-сценария.
- **Этап 4. Полный скоринг → топ 5–7.**
- **Этап 5. Дебаты** (дорого — только для лучших). Разбитые выбывают; устоявшие получают вероятность судьи.
- **Этап 6. Выдача топ-3** + отчёт воронки: что сканировали, сколько и почему отсеяно на каждом этапе (прозрачность).

Правила дисциплины: <30 кандидатов = «слабый день», легитимен ответ «стоящих идей нет» (лучше, чем натянутая идея); пустая выдача после этапа 5 — тоже результат с журналом причин; диверсификация топ-3 по макро-драйверам; журнал воронки целиком; идея, дважды разбитая судьёй, замораживается на N дней — запрет пересдавать до случайного прохождения.

# ЧАСТЬ III. ЧЕСТНОСТЬ, КАЛИБРОВКА, ЗАПУСК

## 9. Стандарт разрешимости прогноза
Прогноз принимается в журнал, только если однозначен ДО события: актив/контракт, направление, величина, срок, источник цены для сверки. «Нефть вырастет» — не прогноз; «фьючерс Brent ближайшей серии закроется выше $X до даты Y по данным Z» — прогноз. Запечатывание: timestamp + hash содержимого — подделка задним числом невозможна.

## 10. Правила обучения и устойчивой калибровки
1. **Минимальная выборка:** вес меняется после N≥30 сопоставимых разрешённых исходов, никогда — после единичного промаха.
2. **Регуляризация:** шаг изменения веса ≤ ±10% от текущего за итерацию.
3. **Holdout:** часть форвард-исходов никогда не используется для настройки; бюджет обращений — 4/год, каждое обращение журналируется (многократное подглядывание убивает holdout через голову исследователя).
4. **Значимость:** изменение принимается только при значимом улучшении на out-of-sample.
5. **Версии и откат:** каждая конфигурация весов версионируется (weights.yaml + git).
6. **Ежеквартальный ручной аудит:** человек выборочно проверяет 10 судейских решений — рубрика и слепота со временем «протекают» (модели учатся писать под рубрику), это ловится только глазами.
7. **Редкие события:** веса по ним не обучаются — фиксируются априорно с пометкой «неоткалибровано».
8. **Стоп-критерий:** обучение до плато целевой функции на holdout; плато = «достижимая идеальность» (§2).
9. Метрики: Brier по корзинам, hit rate, доходность сверх бенчмарка, max drawdown, предотвращённые ошибки.
10. Промах → разборщик объясняет причину по протоколу прогона (какой голос завёл не туда) → предложение поправки → поправка проходит пункты 1–5.

## 11. Этапы (только денежные ворота), абляция, kill-критерий
Этапность по составу агентов отменена — строится всё сразу. Сохраняются только ворота, отделяющие систему от денег:
- **Этап С (сборка + калибровка на известном):** полный контур всех агентов; класс активов — сырьё как ядро + расширение на акции/опционы сразу после прохождения gate сборки; 2–3 API; фиксированный бюджет прогона. Включает программу §23 целиком.
- **Gate С→Б:** программа §23 пройдена (детекторы и пороги откалиброваны, библиотеки наполнены); сквозной прогон полной воронки со всеми агентами без нарушений П8; маскированные кейсы ≥70% по рубрике; запечатывание работает.
- **Этап Б (бумага):** запечатанные форвард-прогнозы без денег; калибровочный режим набирает статистику мелкими недельными прогнозами. ВСЕ агенты участвуют, абляция работает с первого дня.
- **Gate Б→Д:** 270+ разрешённых прогнозов (значимость при hit rate ~55% против 50%) ИЛИ осознанная письменная подпись пользователя «иду дальше на недоказанной статистике»; калибровка ±10 п.п.; превышение над бенчмарком на бумаге.
- **Этап Д (микроденьги):** 0.5% капитала/идея; Келли — после подтверждения калибровки.
- **KILL-критерий (записан заранее, против sunk cost):** после 270 разрешённых прогнозов нет значимого превышения над бенчмарком ИЛИ калибровка хуже ±15 п.п. → проект закрывается или перезапускается с новой гипотезой edge.

### 11.1. Абляция вместо поэтапного ввода
Раз все агенты живут сразу, вопрос «кто реально приносит пользу, а кто шумит» решается абляцией:
- Дирижёр на каждом синтезе сохраняет в протокол не только итоговую вероятность, но и **контрфактические итоги**: «какой была бы вероятность без голоса агента X» (пересчёт агрегации без X — дёшево, это математика, не повторные LLM-вызовы).
- Ежемесячно по форвард-исходам считается вклад каждого агента: улучшал ли его голос Brier итога или ухудшал.
- Агент с устойчиво отрицательным вкладом (по правилам §10: N≥30, значимость) понижается в весе вплоть до карантина — но НЕ удаляется: его суждения продолжают журналироваться, как у агента примет.
- Это полный эквивалент A/B-ввода по информативности, без растягивания запуска.

## 12. Защита пользователя от себя
- Пре-коммитмент: максимальный риск на идею и на месяц фиксируется письменно заранее; лимиты зашиты В КОДЕ (limits.yaml + проверки оркестратора), не в обещаниях; система отказывается обсуждать превышение в моменте.
- Две корзины: «медленные» идеи (каскады — основная игра, живут днями и неделями) с паузой ≥ суток до решения; «быстрые» — только в пределах 10% риск-бюджета, непополняемого в течение месяца.
- Журнал решений пользователя (принял/отклонил/мотив) → сверка с исходами → пользователю показывается ЕГО собственная калибровка.
- После 3 убыточных принятых идей подряд — автоматический режим «только бумага» на 2 недели.
- Анти-комфорт: профиль компетенции обновляется только по подтверждённой правоте (реально разбирался и был прав), не по «понравилось» — иначе круг компетенции выродится в круг комфорта, и система научится подсовывать приятное вместо верного. Раз в квартал — принудительная порция идей вне круга с пометкой.
- Ежеквартальный «отчёт о мышлении»: где экспертиза пользователя реальна, где иллюзорна; на каких типах идей он систематически прав/неправ.

## 13. Круг компетенции (USER_CONTEXT.md)
Приоритетные области (дополняется): полезные ископаемые и сырьё; сервисные компании; цифровые процессы и IT; искусственный интеллект; автомобили; космос и запуски ракет; растущие «компании будущего», меняющие мир бизнес-модели. Вне компетенции: узкоспециальное (пример: цветоводство). Идеи вне круга штрафуются (вес 10% в скоринге), но проходят при исключительном балле.

## 14. Исполнение и активы
- Универсум: ликвидные фьючерсы и ETF на сырьё, крупные акции, опционы на них. NFT исключены (неликвид, манипулируемость, нет данных, нет шорта). Фильтр ликвидности: наш объём незаметен в стакане; тонкие рынки запрещены.
- **Карантин сенсаций по классам:** сенсационная новость не порождает рекомендацию до второго независимого подтверждения ИЛИ истечения окна: ликвидные макро-активы — 2–6 часов; отдельные акции — до 24 часов; малые/экзотические активы — до 48 часов. Скорость сознательно обменяна на достоверность: наша игра — каскады, не фронтран.
- Никогда не входить по рынку в момент всплеска — лимитные заявки, вход частями по уровням.
- Стопы не на очевидных уровнях (круглые числа, минимумы дня — место охоты); смещённая стоп-логика либо замена стопа опционной защитой с учтённой стоимостью.
- Журнал ловушек: каждая распознанная И пропущенная манипуляция → библиотека паттернов → обучение детекторов и разбор промахов.

## 15. Наблюдаемость
Дашборд: калибровочная кривая по корзинам; hit rate и P&L по школам/источникам/типам идей; «стоимость инсайта» (затраты на прогон/идею); статус воронки; журнал обращений к holdout; реестр версий весов и pinned-моделей; счётчики предотвращённых ошибок.

## 16. Реестр неустранимых рисков (открыто, в каждом квартальном отчёте)
1. Edge может не существовать → kill-критерий, дешёвая проверка бумагой и микроразмером.
2. Загрязнение знаний LLM будущим → pinned-версии, прогнозы только вперёд, §23.
3. Нестационарность рынка → приоритет «вечных» причинных связей над хрупкими паттернами; постоянная переоценка.
4. Человеческий фактор: пользователь может сломать любую защиту → лимиты в коде + зеркало его калибровки.
5. Стоимость владения может съесть прибыль → явный бюджет, минимум источников, метрика «стоимость инсайта».
6. Деградация суда: модели учатся «писать под рубрику», релизы моделей ломают воспроизводимость → pinned на квартал, версионируемая рубрика, ежеквартальный ручной аудит (§10.6).

## 17. Режимы работы
1. **Свободная генерация:** полный цикл воронки «найди лучшие неочевидные идеи сейчас».
2. **Тематический:** выбранная тема/актив ведётся изо дня в день: события 1–4 порядка → консенсус → запечатанный прогноз → ежедневная сверка.
3. **Калибровочный:** массовые мелкие недельные прогнозы для набора статистики (без ставок).
4. **Исторический:** наполнение библиотек (взаимосвязи, лаги, предвестники) и калибровка детерминированных компонентов по §23 — НЕ для оценки качества LLM-прогнозов.
5. **Разбор:** еженедельная сверка исходов и разбор промахов.
Всё живёт в одном проекте; журналы накапливаются между запусками.

# ЧАСТЬ IV. ПРОГРАММА КАЛИБРОВКИ И ПЛАН ЗАПУСКА

## 18 (§23). Программа калибровки на известных данных
Разрешение конфликта «хотим учиться на истории» ↔ «LLM помнит будущее»: система состоит из двух типов компонентов, и для них правила противоположны.

### 23.1. Детерминированные компоненты — Честная зона: полный walk-forward на истории
Код не «помнит будущее», поэтому классическая схема (окно обучения → окно проверки → сдвиг окна) полностью легальна для:
1. **Пороги тайминг-детектора:** калибровка «1.5σ» и порогов объёма/OI на истории — при каком пройденном ходе продолжение в среднем ещё было/уже не было.
2. **Лаги переноса каскадов:** измерить на истории фактические задержки (засуха→пшеница: сколько недель; ставка ЦБ→ипотека→девелоперы: сколько месяцев) → knowledge/causal_links с эмпирическими лагами и доверительными интервалами.
3. **Библиотека предвестников (Историк-2):** по 30–50 крупнейшим историческим движениям выбранного класса активов — какие измеримые события им предшествовали; частота, с которой предвестник реально предшествовал движению vs ложные срабатывания.
4. **Детекторы манипуляций (ценовые):** разметка исторических ложных пробоев/охот за стопами → подбор параметров детектора с walk-forward-проверкой.
5. **Статистическая проверка примет и циклов:** каждая примета/цикл прогоняется по истории с FDR-поправкой; выжившие получают статус «кандидат» (окончательно — после форварда).
6. **FDR-порог скана сигналов (§6):** фоновые дисперсии частот слов и метрик считаются на истории.
7. **Поведенческие прокси:** проверка на истории, что выбранные прокси (потоки розницы, плечо, тональность) реально коррелировали со стадиями циклов с опережением, а не запаздыванием.
8. **Бенчмарк и издержки:** исторические спреды, комиссии, проскальзывание — для честного net-матожидания.

### 23.2. LLM-компоненты — Запретная зона и три легальных обходных пути
Запрещено: подавать LLM «точку в прошлом» и оценивать её «прогноз» — модель помнит исход (попросишь спрогнозировать март 2020 — она «угадает» обвал). Легально:
- **(а) Пост-cutoff окно:** точки строго после cutoff самой свежей используемой модели; даже здесь — пометка «возможна утечка через дообучение», результаты считаются ориентировочными, не доказательными.
- **(б) Маскированные кейсы:** исторические ситуации с заменой имён, дат, стран и чисел на нейтральные — проверяется не знание исхода, а качество РАССУЖДЕНИЯ (строит ли агент правильные каскады, задаёт ли вопрос «кто продаёт»). Оценивается логика по рубрике, не точность прогноза. Ограниченно: маскировка несовершенна.
- **(в) Форвард-калибровка (главный путь):** калибровочный режим — десятки мелких разрешённых недельных прогнозов с первого дня этапа Б; они быстро наполняют корзины Brier без единой ставки.

### 23.3. Последовательность калибровки (внутри этапов С–Б)
Данные и котировки → mathlib (тесты) → 23.1 пп.6,8 (фон и издержки) → 23.1 пп.1–5,7 (детекторы, лаги, библиотеки) → сборка воронки с равными весами голосов → 23.2(б) маскированные кейсы как дымовой тест рассуждений → запуск 23.2(в) форвард-калибровки → меритократические веса включаются по мере накопления форвард-статистики (§10).

## 19 (§24). План полной сборки с критериями
Порядок диктуется инженерными зависимостями, не урезанием состава: данные → математика → калибровка на известном → все агенты → состязательный контур → боевой запуск.
| Шаг | Содержание | Критерий перехода (gate) |
|---|---|---|
| Нед. 1 | Каркас репо (§28), config, ответы на §30, подключение котировок | Котировки качаются, история ≥ 10 лет по сырью и акциям |
| Нед. 2 | 2–3 новостных API, сборщик, скрейпер трендов, сантимент на дешёвой модели | Суточный поток нормализуется и тегируется без ручной правки |
| Нед. 3 | mathlib целиком: Brier, FDR, запечатывание, сверка, индикаторы теханализа, разметка волн, Келли, лимиты; юнит-тесты | Все тесты зелёные; прогноз нельзя изменить после записи |
| Нед. 4 | §23.1 полностью: фоновые дисперсии, издержки, пороги тайминга, детекторы манипуляций, лаги, первые 30 причинных связей, библиотека предвестников по 30–50 движениям | Walk-forward-отчёты по всем детекторам; обе библиотеки наполнены |
| Нед. 5–6 | Промпты ВСЕХ агентов блоков B–D и G; оркестратор воронки; OpenRouter с фолбеками; Дирижёр с контрфактическим протоколом (§11.1) | Сквозной прогон: все школы выдвигают кандидатов, поле суждений собирается в стандартном формате |
| Нед. 7 | Состязательный контур (слепота, рандомизация, рубрика), риск-агент, портфельный менеджер, скоринг, синтезатор | Полный цикл этапов 1–6 воронки на тестовом дне |
| Нед. 8 | Маскированные кейсы §23.2(б); дашборд; абляционная отчётность | ≥70% кейсов: рассуждение по рубрике без нарушений П8; дашборд показывает все метрики §15 |
| Нед. 9 | Первый боевой прогон (режим 1); запуск калибровочного режима 2 р/нед; тематический режим по активу №1 | Отчёт §8 со всеми 13 полями; первые запечатанные прогнозы; Gate С→Б пройден |
| Нед. 10+ | Этап Б: бумага; еженедельный разбор; ежемесячная абляция | Gate Б→Д по §11 (270 прогнозов или подпись) |
Бюджет токенов фиксируется на каждый шаг заранее (limits.yaml); превышение — стоп и разбор. Полный контур дороже поэтапного: ориентировочно ×2–3 по токенам на прогон — закладывается в бюджет §30 сразу.

## 20 (§25). Единая петля «проверка → балансировка → обучение»
| Период | Что происходит |
|---|---|
| Каждый прогон | Валидатор (П8) + протокол Дирижёра + проверка лимитов кода |
| Ежедневно (тематический режим) | Сверка вышедших сроков прогнозов; обновление листа ожидания триггеров |
| Еженедельно | Режим разбора: исходы → Brier по корзинам → объяснения промахов → ПРЕДЛОЖЕНИЯ поправок (не применение!) |
| Ежемесячно | Калибровщик применяет накопленные поправки по правилам §10 (N≥30, шаг ≤10%, значимость); версия весов фиксируется |
| Ежеквартально | Ручной аудит 10 судейских решений; «отчёт о мышлении» пользователю; обновление pinned-моделей с регрессионной проверкой на маскированных кейсах; квартальная порция идей вне круга комфорта; обновление реестра неустранимых рисков |
| 2 раза в год | Обращение к holdout (из бюджета 4/год): подтверждение, что система улучшилась, а не подогналась |
Разделение «предложить поправку» (еженедельно) и «применить» (ежемесячно, по правилам) — главный механизм против блуждания настроек.
