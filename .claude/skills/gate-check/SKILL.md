# Проверка gate (MASTER_SPEC §24, §11)

1. Определи текущий шаг по ops/goals.md (выполненные отмечены).
2. Прогони критерий шага БУКВАЛЬНО по чек-листу §24: тесты, наличие файлов, объёмы библиотек, работоспособность потоков. Каждый пункт — командой или скриптом, не «на глаз».

## Этапы ДОРОЖНОЙ КАРТЫ ФИКСОВ (Fn, раздел в ops/goals.md) — ОБЯЗАТЕЛЬНЫЙ цикл ревью
Для шагов с префиксом `F` gate НЕ ПРОЙДЕН без ВСЕХ четырёх пунктов (нулевая автономия):
- (а) **тесты зелёные** — полный `pytest -q` (пред-сущ. провалы помечать явно, новых нет);
- (б) **ревью этапа ПРОЙДЕН** — `Workflow {name:"stage-review", args:{stage:"Fn", base:"<тег/коммит начала этапа>"}}`; вердикт `gate=="ПРОЙДЕН"`, blockers=[], нет регрессий. НЕ закрывать этап при blocker/регрессии — сначала починить;
- (в) **поток подтверждён** — перепрогон `event_first` (mock или live по ситуации): суд не валит 100% и идеи доходят дальше, чем до этапа (`ideas_flow_ok` из stage-review это и оценивает);
- (г) слой Б (FГ) — только при письменной подписи владельца в журнале решений (пороги/рубрика/§10).
Вердикт: ПРОЙДЕН → git tag `fix-Fn`, отметь `[x]` в ops/goals.md, переходи к следующему /goal. НЕ ПРОЙДЕН → точный список блокеров.
3. Денежные ворота §11: Б→Д требует 270 разрешённых прогнозов ИЛИ письменной подписи пользователя в журнале решений; калибровка ±10 п.п.; превышение бенчмарка (60% SPY + 40% BCOM) на бумаге.
4. Вердикт: ПРОЙДЕН (отметь в ops/goals.md, предложи git tag gate-<имя>) / НЕ ПРОЙДЕН (точный список недостающего).
5. Kill-критерий: если 270 прогнозов набраны и превышения над бенчмарком нет ИЛИ калибровка хуже ±15 п.п. — обязан прямо сказать: «сработал kill-критерий §11, по утверждённой спецификации проект закрывается или перезапускается с новой гипотезой edge». Не смягчать.
