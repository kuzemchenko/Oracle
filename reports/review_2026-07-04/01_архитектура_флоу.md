# Ревью архитектуры и флоу «Оракул» (read-only, 2026-07-04)

## Сводная оценка

**Сильные стороны.** Архитектура зрелая и дисциплинированная: детерминированная математика действительно вынесена в mathlib/ с тестами; запечатывание — hash-chain + верификация перед сверкой; денежный трек fail-closed (деньги только при явном «УСТОЯЛА» слепого суда, F0#2/F0#3 — двойная защита в `_money_kind` и `seal_spec`); состязательность П10 обеспечена КОДОМ, а не конфигом (пост-вызовные семейства, обезличивание A/B/C, пересчёт вердикта кодом); бюджет — двухэшелонный (precheck + RunBudgetGuard на BaseException). Протоколы прогонов исчерпывающие, отсев прозрачен по этапам.

**Слабые стороны.** (1) Хрупкость live-пути event_first: несколько мест, где отказ LLM/ключа роняет весь дневной прогон целиком и молча (без протокола → бот молчит). (2) Нулевая межпроцессная синхронизация: cron, systemd-бот и ручные запуски пишут в одни журналы (predictions.jsonl, oracle.db) без file-lock. (3) Дублирование логики между слоями (два скана с разными нуль-моделями; зеркало money-гейта в боте). (4) God-модули: funnel.py (805), bot_reports.py (1121), event_first.py (694) — оркестрация, математика-склейка и рендер перемешаны.

## 1. End-to-end путь event_first: разрывы и потери

**[HIGH] Картограф без обработки отказа LLM роняет весь дневной прогон — orchestrator/event_mapping.py:94 (`propose_cascade`) + event_first.py:466.**
`client.complete(...)` в `propose_cascade` не обёрнут в try: при исчерпании всей цепочки фолбеков роли (`LiveClient.complete` → `RuntimeError`, openrouter.py:218) исключение пролетает через `map_cluster` → `proposal_chains` → `run_event_first`, где ловится только `RunBudgetExceeded`. Сценарий: инцидент OpenRouter/429 по всем трём моделям роли generator в 9:00 → cron-прогон умирает трейсбеком, протокол `ef_*.json` НЕ пишется → бот ничего не пушит → владелец не узнаёт о сбое («тихий отказ»); гибнет даже детерминированная часть (авторские каскады, граф, seal), которой LLM не нужен. Рекомендация: per-cluster try/except (как в `_vet_money`), сбой картографа → пустой список карт + честная пометка в протоколе.

**[HIGH] `_vet_money` создаёт LiveClient вне try — event_first.py:353.**
`client = LiveClient(run_id=run_id)` бросает `RuntimeError`, если `OPENROUTER_API_KEY` не задан, и это не перехвачено. Сценарий: cron идёт с mode="auto"; .env побился/ключ отозван → скан и (mock-)картограф отработали, а на этапе суда прогон падает целиком — снова без протокола и без уведомления. Плюс несогласованность: везде клиент через `make_client(mode)`, тут — жёстко Live. Рекомендация: фабрика + graceful-ветка «суд недоступен → все money демотируются с причиной» (это уже согласуется с fail-closed `_money_kind`).

**[HIGH] Доставка в бот теряет идеи при сбое отправки — ops/bot.py:716–796 (`_tick_reports`).**
`self.state["pushed_runs"].append(run_id)` выполняется независимо от успеха `send_message` (клиент `Telegram._call` при любой ошибке возвращает `None`, это нигде не проверяется). Сценарий: сеть/лимит Telegram в момент пуша money-карточки → карточка не дошла, но run_id помечен как отправленный → идея денежного трека (поверхность §11/§12) потеряна для пользователя навсегда, повторной доставки нет. Рекомендация: помечать run_id пушнутым только после успешной отправки (или пер-карточный статус доставки с ретраем в следующем tick).

**[MEDIUM] Мёртвая ветка вывода в run.py:194–198.**
`cr = src.get("каскад_резолв") or {}` — протокол event_first такого ключа больше не содержит (per-source резолв удалён; `resolve_cascade` жив только в event_first_dryrun.py:94). Консоль всегда печатает «каскад §9: —», вводя оператора в заблуждение, что запечатываний нет. Аналогично `контур_выдал_топ3` под боевым `--vet` всегда пуст (задокументировано, но остаётся легаси-шумом).

**[MEDIUM] Счётчик «демотировано_судом» лжёт, money-пропускная способность скрыто = 3/день — event_first.py:558–568.**
Судятся только `треки["money"][:vet_money_k]` (топ-3), но запечатываются ВСЕ money-узлы: для узлов ниже топ-3 `суд_money.get(sym)` → None → провизорный + `запечатано["демотировано_судом"] += 1`, хотя суда не было. Метрика протокола/дайджеста завышает «разбитость» судом (П8-честность отчётности) и маскирует то, что money-узел №4 структурно не может стать money. Рекомендация: отдельный счётчик «не судился (за пределами top-K)».

**[MEDIUM] Нет идемпотентности запечатывания между прогонами — event_first.py:559–574.**
Дедуп тикеров есть только внутри прогона (graph_select.select). Повторный запуск той же команды в тот же день (ручной перезапуск после сбоя, случайный дубль cron) породит идентичные прогнозы (тот же threshold=close, тот же resolve_by) → коррелированные дубли в предикшн-журнале → искусственное «дозревание» гейта-270 и искажение Brier. Плюс провизорный трек запечатывает ВСЕХ выживших ворот (не top_k) каждый день — персистентное событие ре-запечатывается ежедневно, псевдорепликация провизорного Brier. Рекомендация: дедуп «asset+direction+resolve_by(+threshold)» перед seal.

**[LOW] Ранжирование кандидат-событий смешивает несравнимое — event_scan.py:124.**
Ключ `(салиентность or 0, -(q_value or 1.0))` ставит любой новостной кластер (salience≥1) выше любого статзначимого ценового сигнала. На выдачу влияет слабо (шок-источники собираются отдельно), но «топ_события» в протоколе/дайджесте системно news-biased.

**[LOW] Потеря инструментов картографа — bot_reports.py:213–219 и event_first `_proposal_ideas`:** в поток идёт только `insts[0]`, остальные верифицированные тикеры узла теряются из выдачи (частично признанный F1-low долг).

В остальном сшивка сквозная и без потерь: скан → FDR (единый по цене+трендам, честная пометка про news) → шок-источники → картограф один раз (карты питают и граф, и стейджинг — дедуп) → activated_chains (широкая активация) → build_from_db (F0#8 adjusted, F3#25 asof-гейт) → node_to_facts → ворота/пред-ранг → треки → vet → seal → протокол → бот (дайджест + money-карточки с гардом идемпотентности по токену).

## 2. Устойчивость LLM-слоя (openrouter.py)

Хорошо: 429/5xx/пустой контент → `_Retryable` → следующий фолбек; каждая смена модели журналируется (`log_model_swap`); неуспех пишется в costs.jsonl с ok=false; `agents.call_agent` ловит всё и возвращает запись-брак (воронка fail-soft — П8-вето вместо крэша); `RunBudgetExceeded(BaseException)` не глотается широкими except.

**[MEDIUM] Нет ретраев/бэкоффа внутри модели — openrouter.py:195–218.** Один транзиентный 429 сразу уводит на другое семейство. Для П10-чувствительных ролей это меняет фактические семейства (код это корректно учитывает пост-вызовно, F3#23), но при массовом rate-limit цепочка исчерпывается за секунды без единой повторной попытки. Рекомендация: 1–2 ретрая с бэкоффом на `rate_limit` перед фолбеком.

**[MEDIUM] Направление fail-open/closed в целом правильное**, но асимметричное: сбой судьи → ВЕТО/провизорный (fail-closed, верно); сбой всей роли в funnel-поле → идея просто без голоса (приемлемо); сбой картографа → крэш прогона (см. HIGH выше) — единственное место, где fail-closed вырождается в fail-total.

**[LOW] `usage.cost` может отсутствовать** → в costs.jsonl `cost_usd: null` (П8-честно), но precheck-оценка и месячный спенд тогда систематически занижены — тихая дыра бюджета при провайдерах без цены.

## 3. Связность модулей

- **Циклов импортов нет**: единственный потенциальный (event_scan → multi_event → funnel) развязан ленивым импортом (event_scan.py:94) — осознанно, задокументировано.
- **[MEDIUM] Дублирование скана**: funnel.stage1_scan (funnel.py:171–205) считает p-value нормальным `math.erfc` прямо в оркестраторе, тогда как event_scan после F2#19 перешёл на тяжелохвостый Стьюдент-t в mathlib/tailprob. Два расходящихся FDR-скана одной сущности: тематический/idea-check путь (`run_funnel`, «разбери ТИКЕР» из бота) до сих пор пропускает шум щедрее, чем боевой. Заодно это нарушение «математика — в mathlib».
- **[MEDIUM-LOW] Зеркало money-гейта**: `bot_reports._money_survived` (bot_reports.py:131) вручную зеркалит `event_first._money_kind` (event_first.py:217). Дрейф одной из копий → бот покажет как money то, что контур демотировал (или наоборот). Рекомендация: импортировать одну функцию.
- **God-модули**: funnel.py (805: этапы 1–6 + рендер Markdown), bot_reports.py (1121: контракт карточек + вся презентация), event_first.py (694). Рендер стоило бы отделить от оркестрации.

## 4. Соответствие инвариантам

- **«Дирижёр не имеет мнения»** — выдержано: run.py/funnel.py — маршрутизация, поле, карта противоречий, контрфакты, процедурное вето; агрегат — детерминированное среднее. Долги (задокументированы): меритократические веса §5.3 и экстремизация §5.5 не реализованы. Пограничное [LOW]: ops/bot_chat.py даёт Дирижёру LLM-голос (роль conductor, anthropic) для обсуждения идей с владельцем — заземлено и с рамкой, но это фактически «мнение о рынке» словами Дирижёра.
- **Воронка §6** — реализована честно (все 6 этапов, журнал отсева, «идей нет — легитимно»), масштабы канона (200–500 сигналов, 30–50 кандидатов) не достигаются и честно помечены; боевой контур — амендированный event_first. Замечание [LOW]: «заморозка идеи, дважды разбитой судьёй, на N дней» (§6) в коде не найдена.
- **П10 состязательность — реально обеспечена кодом**: models.yaml разносит семейства (генератор anthropic / критик openai / судья google); `filtered_chain` исключает семейства из всей цепочки фолбеков с RuntimeError при пустой цепочке; debate.py берёт семейства ПО ФАКТУ ответившей модели (F3#23), критик исключает семейство генератора, судья — семейства всех трёх дебатёров; слепое дело A/B/C с детерминированным seed, ревьюер видит обезличенное; вердикт пересчитывается кодом из рубрики (`_adjudicate`), обязательные вопросы — процедурное вето. Остаточный нюанс (осознанный): адвокат может совпасть семейством с генератором — обоснование в коде корректное.
- **Математика в mathlib** — в основном да (Brier, FDR, Келли, каскад, sealing, outcomes). Исключения: funnel.stage1_scan (erfc-p-values в оркестраторе, см. выше), мелкая склейка (sigma_h в graph_build) — приемлемо.

## 5. Операционный контур

- **cron/systemd** — грамотно: явный подъём .env, абсолютный python, коды выхода, Restart=always у бота, бот не в crash-loop без токена, baseline против спама бэклогом.
- **[HIGH] Нет межпроцессного лока на journal/predictions.jsonl — mathlib/sealing.py:85–105.** `seal()` = «прочитать весь файл ради `_last_hash` → append»; `_LOG_LOCK` в openrouter — только threading (один процесс). Сценарий: cron event_first запечатывает в 09:0x, владелец в этот момент жмёт /calibrate в боте (отдельный процесс, тоже seal) → обе записи получают одинаковый prev_hash → verify_all навсегда помечает вторую битой → resolve.py:88 никогда не вынесет по ней исход (перманентная потеря прогноза + вечный «журнал_цел=false»). Рекомендация: fcntl.flock вокруг read-last+append (или лок-файл прогона).
- **[MEDIUM] Два параллельных прогона**: (а) precheck бюджета не атомарен — оба пройдут проверку, суммарно перекрыв месячный потолок (RunBudgetGuard ограничивает только каждый в отдельности); (б) конкурентная запись в oracle.db (`ensure_history` из cron-прогона vs бот-прогона) без `timeout=` у sqlite3.connect → возможен «database is locked» и крэш одного из прогонов. Внутри бота параллельность честно запрещена (`_job_busy`), но бот-против-cron ничем не защищён.
- **[MEDIUM] Тишина при падении cron-прогона**: единственный след — journal/cron.log; бот уведомляет только по появившимся протоколам. Рекомендация: wrapper, шлющий алерт при ненулевом коде выхода 09:00-задачи.
- **[LOW] Гигиена**: cron.log/bot.log без logrotate; `pushed_runs`/`seen_watchlist` в bot_state растут неограниченно; verify_all перечитывает весь журнал на каждый seal — O(N²) по мере роста (пока 409 строк — не больно).
- Идемпотентность: resolve — идемпотентен (дедуп по hash); event_first — НЕ идемпотентен по seal (см. §1).

## Статус известных багов (все 4 подтверждены ОТКРЫТЫМИ)

1. **Hash-цепь не ловит усечение хвоста** — mathlib/sealing.py:130–148: `verify_all` проверяет только внутреннюю согласованность; удаление последних N строк оставляет валидную цепочку (нет внешнего якоря последнего hash/счётчика).
2. **portfolio.py:174** — `lim.check_monthly_risk(0.0, total)`: накопленный месячный риск захардкожен 0.0, месячный лимит $3000 фактически проверяет только текущий прогон.
3. **debate.py:215** — протокол возвращает `"направление": direction` (сырое, None у masked/каскадных без школы) вместо `eff_direction`, которым реально велось дело.
4. **cascade.py:216–218** — `_lag_for_pair`: сравнение `{a,b} == set(pair)` направленно-агностично, лаг A→B применяется и к B→A.

**Приоритет починки:** межпроцессный лок seal + доставка бота (HIGH №3–4) и крэш-пути картографа/vet (HIGH №1–2) — раньше всего: они бьют по обеим главным ценностям системы (честность журнала §16/П16 и доведение идей до владельца).
