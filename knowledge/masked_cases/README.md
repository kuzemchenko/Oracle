# Маскированные кейсы — §23.2(б) MASTER_SPEC

Регрессионный набор для проверки **качества РАССУЖДЕНИЯ** агентов (не угадывания исхода).
Запускается `orchestrator/masked.py` (или `python3 orchestrator/run.py --mode masked`).

## Зачем (§23.2(б), §16.2)
LLM «помнит будущее»: дать модели точку в прошлом и оценить её «прогноз» — нечестно (инвариант 3
CLAUDE.md, П16). Легальный обходной путь — **маскировка**: в исторической ситуации заменяем
**имена, даты, страны и числа** на нейтральные. Проверяем не знание исхода, а строит ли агент
правильные каскады (1→2→3 порядок), задаёт ли вопрос «кто продаёт нам», учитывает ли тайминг и
манипуляцию, разрешимость §9 и net-асимметрию — то есть 6 критериев `config/rubric.yaml`.

## Что отдаётся агентам, а что нет
Каждый кейс делится на два блока:
- **`подаётся_агентам`** — ТОЛЬКО это видят генератор/критик/адвокат/reviewer/судья: маскированная
  ситуация + нейтральные числа/индикаторы. Ни исхода, ни ожидаемого направления, ни оригинального
  события здесь нет.
- **`эталон_АУДИТ`** — эталон рассуждения, ожидаемое направление, подсказка об оригинале. Только
  для человека и ежеквартального аудита (§25). Гарантированно НЕ передаётся ни одному агенту
  (см. `orchestrator/masked.py::_agent_payload` — берёт строго `подаётся_агентам`).

## Несовершенство маскировки (честно, П8)
Маскировка несовершенна: родовые связки («засуха у крупного экспортёра → рост агро-фьючерса»)
модель может опознать как КЛАСС, не опознав конкретный эпизод (даты/страна/числа нейтрализованы).
Каждый кейс несёт поле `несовершенство_маскировки`; харнесс помечает результат флагом
`маскировка_несовершенна=true` и не выдаёт результаты за доказательные (только ориентировочные).

## Оценка (детерминированно, mathlib)
`mathlib/masked_eval.py` (не LLM, инвариант 6) по каждому кейсу:
- средний балл рубрики судьи (0–5) → процент от максимума;
- кейс **зачтён**, если средний балл ≥ `verdict.break_threshold` (3.0/5) И нет нарушений П8 во
  всей цепочке ролей И отвечены оба обязательных вопроса §4 (кто продаёт / почему ещё существует);
- **gate §24**: доля зачтённых кейсов ≥ **0.70** (спека: «≥70% кейсов: рассуждение по рубрике без
  нарушений П8»).

## Регрессия при обновлении моделей (§25, §16.6)
Набор фиксируется в git. При ежеквартальном обновлении pinned-моделей (`config/models.yaml`
`meta.pinned_quarter`) прогон повторяется: падение доли зачтённых ниже 0.70 — сигнал, что новые
веса моделей сломали качество рассуждения. Состав кейсов меняется только осознанно (как рубрика).
